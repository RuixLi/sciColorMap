# Tree-graph filesystem

The model the framework's documents are organized by: a **tree** for containment and addressing, overlaid with a **graph** of links between documents - where every leaf is authored and self-describing, and every summary view is *derived* from the leaves, never authored. This is the reference `where-files-live`, `tracking-progress` lean on; it is a conceptual shape, not itself a checkable rule.

## The tree - containment and addressing

- **The file is the unit of identity.** A document is addressed by its path; name it to identify itself (the `naming-and-formatting` reference). A sub-part is addressed by an in-file section (a heading), not a new file; reserve explicit IDs for in-file anchors.
- **Concerns branch as they grow.** A thin concern is a file `X.md`; a dense one is a folder `X/` of leaves with `X.md` as its index. The entry path stays stable as the inside grows, so links do not churn.

## The graph - links between documents

- **Documents link, and the links carry meaning** - a dependency, a `supersedes` edge to an archived original, a report's next-action, a mode handoff.
- **Every link resolves and reciprocates sometimes** - a forward link names a document that exists (or is about to), and that document cites back if necessary. A forward link with no back-reference is a half-built edge; the graph is no longer traversable both ways (sometimes causes unreachability).

## Leaves self-describe; summaries are derived

- **Each indexed leaf carries a structured header** - a small `YAML` frontmatter bearing the leaf's index-relevant metadata (shape: [document-frontmatter](document-frontmatter.md)). The header is the single source of truth for what the leaf *is* to generated indexes.
- **An index is generated by scanning headers, never authored** - an index (usually `X.md` for a folder `X/`) is *derived* from the leaves' headers, the way `index-artifacts` builds `artifacts.json` from artifact sidecars (`artifact-manifest`). Because it is derived it cannot drift from its leaves.
- **Roles never flip.** A leaf is authored; an index is derived - regardless of how many leaves there are.

## Derived sections are fenced

A leaf may also carry a **tool-owned generated section** - a status line, resolved back-links - but it must be a clearly fenced block, never hand-edited and idempotently regenerable, so authored body and generated section never blur.

## What it implies

- **`where-files-live`** lays the tree and owns path conformance; **`naming-and-formatting`** supplies the address surface forms; **`how-to-work-here`** owns the `supersedes` edge; **`work-documents`** names the leaves and their derived indexes; **`tracking-progress`** runs the derived indexes and authored `jobs`; **`gradient-propagation-workflow`** owns the consistency invariant.
- The **link validator** (reciprocity) and the **sync / indexer** (generate indexes from headers) enforce it; the **artifact-manifest** sidecar -> index is the same shape for data.
