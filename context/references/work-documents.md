# Work documents

The vocabulary of documents a work produces: what each document is for, where it sits in the model, and how the roles differ. The full dynamics are in `gradient-propagation-workflow`; the obligations are in `how-to-work-here`; the procedure is in the run-the-work skill; document shapes are `writing-for-the-user` plus the templates.

## Scope

The set of documents an agent writes while delivering or discussing work, in both archetypes. Referenced by the conventions that produce or surface them (`how-to-work-here`, `tracking-progress`).

Out of scope: the workflow model (`gradient-propagation-workflow`); the workflow obligations (`how-to-work-here`); the format of any document (`writing-for-the-user` and the templates); package release records such as `CHANGELOG.md`.

Obligations about when to create, update, and validate these documents live in conventions and skills. This reference only keeps the terms and distinctions in one place. The names are **universal**: the archetype changes a document's content (a study's oracle is a validation; a package's is a test), never its name.

## The set

Each name points to one role.

| Document | Role - what it holds | Usual point in the work |
| --- | --- | --- |
| **scheme** | the coordinator: decomposes the goal into **milestones** (sequential) and **modules** (parallel), each carving out efforts | once, up front; amended by exception |
| **effort** | a unit of execution; surfaces its assumptions, decisions, and **oracle(s)** - the up-front, independent success check; rolls | per unit, before its first plan; extended slice by slice |
| **plan** | one increment's execution route: the internal architecture and ordered, tests-first tasks | per increment, before execution |
| **log** | the trail: what happened and what changed, in order | appended during execution |
| **report** | a finished increment's outcome, by the **oracle's verdict** (not whether the plan ran): status, related files, next action(s) | each increment finishes |
| **index** | the generated, mechanical index of work-document frontmatter, including report status values | synced from work documents |
| **jobs** | the authored worklist: one row per in-flight **effort**, its current cycle state, intent and next action | as work proceeds |
| **discussions** | the record of iterative work, in threads | as iterative work proceeds |

Beyond the live set is the **archive**: a superseded scheme, effort, plan, log, or artifact, kept intact, with the replacement carrying a `supersedes` link (below).

## Role distinctions

- **Effort vs plan** - the `effort` is the stable contract that declares the oracle (one per unit, rolls only on an upward redirect); the `plan` is the disposable route that satisfies it (many, dated). Kept apart so a failed route is re-planned without disturbing the oracle, and one effort is advanced by a *sequence* of plans (`gradient-propagation-workflow`).
- **The oracle is the effort's load-bearing section**, not its own file: the concrete, independent, up-front check whose verdict closes or redirects the effort. "Done" means the oracle passes, never that the plan ran.
- **Trail vs outcome vs index vs worklist** - the `log` is the chronological trail; the `report` is an increment's outcome by the oracle verdict; an `index` is the mechanical index of document frontmatter, including report status values; `jobs` is the authored worklist of in-flight efforts.
- **Index (derived) vs jobs (authored)** - work documents are the source of truth; indexes are synced from them by tool and never hand-edited; `jobs` is authored alongside and carries the intent and priorities the projection cannot (`tracking-progress`).
- **Authored vs generated** - the leaves and the worklist are authored (`scheme`, `effort`, `plan`, `report`, `log`, `discussions`, `jobs`); the **gate index files** (`efforts.md`, `plans.md`, `logs.md`, `reports.md`) are generated by indexing their folders, with each report verdict carried by that report's `status` field (`tracking-progress`).
- **Coordinator vs unit** - the `scheme` decomposes a goal into efforts (via milestones / modules) and lives among them; the index points to it, or to the efforts directly when there is none (`where-files-live`).
- **Live chain vs archive** - the current, non-archived documents read as what-is-true-now; a superseded document is archived intact and reached by one `supersedes` hop (`gradient-propagation-workflow`).
- **The report is the boundary object** - a report whose next-action links across exploratory and oracle-driven work *is* the mode handoff; there is no special report type.

## Per-document notes

- **scheme** - the coordinator; milestones are sequential (each builds on the last), modules are parallel (independent), and both carve out efforts.
- **effort** - the oracle-declaring unit at any scale (a function, a module, a pipeline transform); always surfaces assumptions, decisions, and oracle(s); rolls - extend the next slice before its plan. An effort without a declared oracle is not an effort.
- **plan** - one dated increment's route: the internal architecture and ordered, tests-first tasks; disposable - a failed route is re-planned, the effort untouched.
- **log** - the chronological execution trail: steps, changes, decisions, failures, and finish. Superseded logs belong to the archive.
- **report** - the finished-increment outcome, by the oracle's verdict: **status** (*passed* / *blocked* / *redirected* / *escalated*), **related files** (the effort, plan, and results touched), and **next action(s)** (each an action plus a link to the document it acts on). Re-enterable without the log.
- **index** - the derived board: the mechanical index of work-document frontmatter, including report status values, written only by the sync tool, so it cannot desync from its source documents; combined into the generated gate indexes rather than kept as a separate authored file (`tracking-progress`).
- **jobs** - the authored worklist: one row per in-flight effort, its current plan-execute-evaluate-report cycle state (planned -> executing -> evaluated), and what to do next; may reference indexes but does not contain them (`tracking-progress`).
- **discussions** - threads opened, discussed (explore -> narrow -> record), and closed; the document is the record for exploratory work.
- **archive** - a superseded scheme, effort, plan, log, or artifact, kept intact; the replacement carries `supersedes -> <archived-file>`, a checkable backward link (`gradient-propagation-workflow`).
