# Project layout

The concrete catalog of a project's key files and folders - what each is, what it holds, and whether it is **must** (required) or **may** (optional) - that the `where-files-live` convention shapes. Non-normative: the rules live in `where-files-live`; this is the detail to converge a real repo. Fill-in shapes are in [templates.md](../templates.md).

## Key files (every project)

- **`README.md`** *(must)* - the human face: title + one-line purpose, then install / quick start / usage. A study's runs thinner. [template](../templates/project/README.md)
- **`AGENTS.md`** *(must)* - the agent entry point at the root; orients in one read (`building-context`). [template](../templates/project/AGENTS.md)
- **`.gitignore`** *(must)* - the allowlist: git tracks text source and ignores generated artifacts, caches, and large binaries. [template](../templates/project/gitignore)
- **environment manifest** *(must)* - pinned dependencies: `pyproject.toml` (Python), `environment.yml` (conda), or `ENVIRONMENT.md` (language-agnostic). [pyproject](../templates/project/pyproject.toml) · [ENVIRONMENT.md](../templates/project/ENVIRONMENT.md). The manifest is the trusted record and truth of dependencies; a local env (`.venv/`) is disposable and git-ignored.
- **`.git/`** *(must)* - the Git repository. **Never inside an actively-synced folder** (e.g. OneDrive): a synced `.git` corrupts, because its many small files upload mid-write. Solutions: 1) Keep the repo on local disk; 2) junction the heavy data folders in from the synced location; 3) put the `.git` folder outside the synced folder and point to it with `git --git-dir=... --work-tree=...`.
- **`CHANGELOG.md`** *(package, must)* - the release record of a package (Keep a Changelog + SemVer). [template](../templates/project/CHANGELOG.md)
- **`LICENSE`** *(package, may)* - the license text; add at release.

## Key folders (every project)

- **`src/`** *(must)* - reusable code as a namespaced package, called via its namespace, callers depending inward. Python `src/<pkg>/`, MATLAB `src/+<pkg>/`. Never loose at the root.
- **`tests/`** *(must)* - the suite, prefix in name hint its role:
  - **`test_*`** *(must, where code exists)* - test code correctness; the agent's seatbelt (`working-with-code`). Underscore, not hyphen - pytest only discovers `test_*.py`, and a hyphen is not legal in a module name.
  - **`validation_*`** *(must, where a result is relied on)* - validate result trustworthiness, anchored to a known answer (`working-with-data`).
  - **`smoke_*`** *(may)* - check package installs and other miscellaneous functionality.
- **`context/`** *(must)* - the agent's working memory; see below.

## The `context/` folder

`context/` is the framework's heart - the durable working memory that lets an agent, or a fresh agent after compaction, re-enter and continue. It is **tree-structured** (`where-files-live`): each concern is a file `X.md` that branches into `X.md` + `X/` as it grows, so there is always a stable entry to navigate down from.

The work documents (`work-documents` reference; shapes in [templates.md](../templates.md)):

- **`jobs.md`** *(may)* - the authored worklist: one row per in-flight effort, its cycle state, and a pointer. The first thing to read for "where do things stand" (`tracking-progress`). [template](../templates/context/jobs.md)
- **`efforts.md` / `efforts/`** *(must)* - the effort contracts, one per unit: each surfaces its assumptions, decisions, and oracle(s). A **`scheme`** coordinates efforts as **milestones** (sequential) and **modules** (parallel) when the goal is large. [effort](../templates/context/effort.md) · [scheme](../templates/context/scheme.md)
- **`plans.md` / `plans/`** *(must)* - the dated execution plans, one per increment, each a route of tests-first tasks that satisfies an effort's oracle. [plan](../templates/context/plan.md)
- **`logs.md` / `logs/`** *(must)* - the chronological trails, one dated file per oracle-driven increment in `logs/`; `jobs.md` is the progress interface, so `logs.md` is a thin pointer. [template](../templates/context/log.md)
- **`reports.md` / `reports/`** *(must when an increment finishes)* - finished-increment outcomes by the oracle's verdict: status, related files, next action(s); re-enterable without the log. [template](../templates/context/report.md)
- **`discussions.md` / `discussions/`** *(may)* - the record of exploratory work, in threads: open -> explore / narrow / record -> close (`how-to-work-here`). [template](../templates/context/discussions.md)

The **gate index** for each folder above (`efforts.md`, `plans.md`, `logs.md`, `reports.md`) is generated by scanning the leaves' YAML headers (`tree-graph-filesystem`), not authored, and exists from the first leaf - no thin authored phase; report verdicts are carried by each report's `status` field (`tracking-progress`).

Other context:

- **`decisions/`** *(may)* - the project's ADRs: durable decisions with context, options, consequences, and status - the rationale future work must preserve.
- **`archive/`** *(may)* - superseded work documents kept intact: a scheme, effort, plan, or log replaced on an upward redirect; the replacement records a `supersedes` link (`how-to-work-here`, `gradient-propagation-workflow`). Never edited or deleted.
- **vendored framework material** *(consuming repo, must)* - the adopted conventions copied in as a hashed dependency: `conventions.md` / `conventions/`, `references.md` / `references/`, `templates.md` / `templates/`; `AGENTS.md` points to `context/conventions.md`. (This repo is the exception - it builds them, so they live at the root.)

**Coordinator vs unit.** A *unit* is one `effort` - the contract that declares an oracle. A *coordinator* is a `scheme` that decomposes a goal into efforts and lives among them, via **milestones** (sequential) and **modules** (parallel), which may coexist. The `efforts.md` index points to the scheme when there is one, or to the efforts directly when there is not - so a scheme and loose efforts can coexist, and a project can hold more than one coordinator.

## Study folders

A study converges on a finding, then archives. Recommended layout:

- **`scripts/`** *(must)* - orchestration that calls `src/`: `process/` (the `raw -> interim -> final` pipeline), `results/` (figures / tables from processed data).
- **`configs/`** *(may)* - project-wide configuration (paths, parameters, sources).
- **`notebooks/`** *(may)* - exploratory analysis; dated subfolders.
- **`data/`** *(must)* - `raw/` (immutable, checksummed) -> `interim/` (regenerable cache) -> `final/` (result-ready).
- **`outputs/`** *(must)* - deliverables: figures, tables, manuscript, slides, models.
- **`references/`** *(may)* - external source material: PDFs, atlases, reference images.
- **`archive/`** *(may)* - deprecated or superseded work.
- **`run/`** *(may)* - stdout/stderr of background runs (`<script>_<timestamp>.log`).

## Package folders

A package converges on a reusable code. Recommended layout:

- **`src/<pkg>/`** *(must)* - the product, fronted by the public API gateway (`__init__.py` / `Contents.m`; the `api-surface` reference).
- **`examples/`** *(may)* - runnable usage; the how-to surface for humans and agents.
- **`docs/`** *(may)* - design notes and generated API docs.
